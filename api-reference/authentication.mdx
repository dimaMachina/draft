The Docs API authenticates every request using API keys passed as Bearer tokens in the `Authorization` header. There are no sessions, cookies, or OAuth flows — each request is independently authenticated, making the API safe to call from any environment including CI pipelines, serverless functions, and backend services.

## Get an API key

You create and manage API keys from the Mintlify dashboard. Follow these steps to generate a new key:

1. Sign in to your [Mintlify dashboard](https://dashboard.mintlify.com).
2. Navigate to **Settings** in the left sidebar.
3. Select the **API Keys** tab.
4. Click **Generate New Key**.
5. Give the key a descriptive name (for example, `ci-deployment` or `content-pipeline`).
6. Select the appropriate scope — `read` or `read_write` — for your use case.
7. Click **Create** and copy the key immediately.

<Warning>
  Your API key is only shown once at the moment of creation. Copy it to a secure location — such as a password manager or a secrets vault — before closing the dialog. If you lose the key, you must revoke it and generate a new one.
</Warning>

## Authenticate requests

Pass your API key in the `Authorization` header of every request using the `Bearer` scheme:

```
Authorization: Bearer YOUR_API_KEY
```

The following examples show how to set this header in common environments:

<CodeGroup>

```bash curl
curl --request GET \
  --url https://api.mintlify.com/v1/pages \
  --header "Authorization: Bearer YOUR_API_KEY" \
  --header "Content-Type: application/json"
```

```javascript JavaScript (fetch)
const response = await fetch("https://api.mintlify.com/v1/pages", {
  method: "GET",
  headers: {
    Authorization: "Bearer YOUR_API_KEY",
    "Content-Type": "application/json",
  },
});

const { data } = await response.json();
```

```python Python (requests)
import requests

response = requests.get(
    "https://api.mintlify.com/v1/pages",
    headers={
        "Authorization": "Bearer YOUR_API_KEY",
        "Content-Type": "application/json",
    },
)

data = response.json()["data"]
```

</CodeGroup>

## Authentication errors

### 401 Unauthorized

The API returns `401` when the `Authorization` header is missing entirely or when the token is malformed or revoked. Check that the header is present and that the key has not been deleted from the dashboard.

```json 401 Response
{
  "error": {
    "code": "unauthorized",
    "message": "Missing or invalid API key.",
    "requestId": "req_7xK2mNpQ"
  }
}
```

### 403 Forbidden

The API returns `403` when the key is valid but does not have the permissions required for the requested operation. For example, a `read`-scoped key attempting to create a page will receive this error.

```json 403 Response
{
  "error": {
    "code": "forbidden",
    "message": "Your API key does not have permission to perform this action.",
    "requestId": "req_9zR4vWsL"
  }
}
```

## API key scopes

Every API key is assigned one of two scopes at creation time. Choose the narrowest scope that satisfies your use case.

| Scope | Permitted operations |
|---|---|
| `read` | `GET` requests only — list pages, retrieve page content, and fetch analytics. |
| `read_write` | All `GET` requests plus `POST` and `PATCH` — create pages and trigger deployments. |

<Note>
  You can create multiple API keys with different scopes and assign them to different services. For example, use a `read`-scoped key for your analytics dashboard and a `read_write`-scoped key for your content pipeline.
</Note>

## Security best practices

Treat your API key like a password — it provides direct access to your documentation project. Follow these guidelines to keep it secure:

- **Never commit keys to source control.** Use environment variables or a secrets manager instead.
- **Rotate keys regularly.** Revoke old keys from the **Settings → API Keys** page and generate fresh ones on a schedule.
- **Use the narrowest scope.** Grant `read_write` only when the client genuinely needs to write data.
- **Monitor for suspicious activity.** Check the API key usage log in the dashboard to detect unexpected spikes.

## Environment variable best practices

Store your API key in an environment variable rather than hardcoding it in source files. The examples below show the most common patterns:

<CodeGroup>

```bash .env file
# .env — never commit this file to source control
MINTLIFY_API_KEY=your_api_key_here
```

```bash Shell export
# Add to ~/.zshrc, ~/.bashrc, or your CI environment
export MINTLIFY_API_KEY="your_api_key_here"
```

```dockerfile Docker
# Pass the secret at run time, not at build time
docker run \
  --env MINTLIFY_API_KEY="your_api_key_here" \
  your-image:latest
```

</CodeGroup>

Then read the variable in your application:

<CodeGroup>

```javascript Node.js
const apiKey = process.env.MINTLIFY_API_KEY;
```

```python Python
import os

api_key = os.environ["MINTLIFY_API_KEY"]
```

</CodeGroup>
